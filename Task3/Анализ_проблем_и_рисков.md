# Анализ текущей архитектуры InsureTech

## Описание текущей архитектуры

### Компоненты системы:
1. **core-app** - монолитное приложение для работы с продуктами и заявками
2. **client-info** - сервис управления клиентскими данными
3. **ins-product-aggregator** - сервис интеграции со страховыми компаниями
4. **ins-comp-settlement** - сервис взаиморасчётов со страховыми компаниями

### Текущие взаимодействия:

**1. core-app - ins-product-aggregator**
- Протокол: REST (синхронный)
- Частота: каждые 15 минут
- Цель: получение актуальных тарифов и продуктов
- Данные кэшируются локально в core-app

**2. ins-comp-settlement - ins-product-aggregator**
- Протокол: REST (синхронный)
- Частота: 1 раз в сутки (ночью)
- Цель: получение информации о продуктах для формирования реестра

**3. ins-comp-settlement - core-app**
- Протокол: REST (синхронный)
- Частота: 1 раз в сутки
- Цель: получение всех оформленных за день страховок

**4. ins-product-aggregator - Страховые компании (5 шт)**
- Протокол: REST/SOAP/GraphQL (синхронный)
- Частота: при каждом запросе от core-app/ins-comp-settlement
- Процесс:
  1. Запрос к ins-product-aggregator
  2. Агрегатор синхронно опрашивает все 5 компаний
  3. Агрегирует результаты
  4. Возвращает в рамках того же запроса

## Выявленные проблемы

### 1. Синхронные вызовы к множеству внешних систем

**Проблема:**
- ins-product-aggregator делает синхронные вызовы ко всем 5 страховым компаниям
- Общее время ответа = max(время ответа каждой компании)
- Если одна компания медленная (5-10 сек) → весь запрос медленный

**Риски:**
- При росте до 10 компаний время ответа может вырасти в 2 раза
- Cascading failures: отказ одной компании блокирует весь процесс
- Timeout errors и retry storms

**Текущее влияние:**
- Пользователи ждут 5-10 секунд загрузки страницы с продуктами
- Периодические ошибки 504 Gateway Timeout
- Плохой UX → снижение NPS

### 2. Tight Coupling (Тесная связанность)

**Проблема:**
- core-app напрямую зависит от доступности ins-product-aggregator
- ins-comp-settlement зависит от доступности core-app И ins-product-aggregator
- Синхронные REST вызовы создают прямые зависимости

**Риски:**
- Если ins-product-aggregator недоступен → core-app не может обновить тарифы
- Если core-app недоступен → ins-comp-settlement не может получить данные о страховках
- Сложность независимого развертывания и обновления сервисов

**Текущее влияние:**
- Простой одного сервиса блокирует работу других
- Невозможность обновлять сервисы независимо
- Сложность тестирования в изоляции

### 3. Polling вместо Push (Опрос вместо уведомлений)

**Проблема:**
- core-app опрашивает ins-product-aggregator каждые 15 минут
- ins-comp-settlement опрашивает сервисы раз в сутки
- Неэффективное использование ресурсов

**Риски:**
- Устаревшие данные: изменения в тарифах становятся доступны с задержкой до 15 минут
- Ненужные запросы: 96 запросов в сутки, даже если данные не изменились
- Нагрузка на сеть и CPU для опроса

**Текущее влияние:**
- Клиенты видят устаревшие тарифы до 15 минут
- Потеря продаж из-за неактуальной информации
- Неэффективное использование ресурсов

### 4. Отсутствие отказоустойчивости при интеграции

**Проблема:**
- Нет механизма повторных попыток (retry)
- Нет circuit breaker для защиты от каскадных сбоев
- Нет fallback на кэшированные данные при ошибках

**Риски:**
- Временные сбои в одной страховой компании приводят к ошибкам для всех пользователей
- Отсутствие graceful degradation
- Накопление неудачных запросов

**Текущее влияние:**
- Ошибки "Сервис временно недоступен"
- Потеря транзакций
- Негативный опыт пользователей

### 5. Масштабируемость: от 5 до 10 страховых компаний

**Проблема:**
- При добавлении 5 новых компаний нагрузка удвоится
- Синхронный подход не масштабируется линейно
- Время ответа будет расти

### 6. Дублирование данных без синхронизации

**Проблема:**
- Локальные реплики данных о продуктах хранятся в:
  - core-app (обновление каждые 15 мин)
  - ins-comp-settlement (обновление раз в сутки)
- Нет механизма гарантированной синхронизации

**Риски:**
- Данные могут быть несогласованными между сервисами
- Тариф изменился, но старый показывается в core-app до 15 минут
- Расчёты в ins-comp-settlement могут быть на основе устаревших данных

**Текущее влияние:**
- Несогласованность данных между системами
- Возможные ошибки в расчётах
- Споры со страховыми компаниями

### 7. Отсутствие истории изменений

**Проблема:**
- Нет записи о том, когда и как изменились тарифы
- Невозможно отследить, какой тариф был актуален в момент оформления страховки

**Риски:**
- Споры с клиентами: "Вы показывали другую цену"
- Проблемы с аудитом и комплаенсом
- Невозможность анализа ценообразования

**Текущее влияние:**
- Отсутствие прозрачности
- Сложность расследования инцидентов
- Риски регуляторных проверок

### 8. Batch обработка данных о страховках

**Проблема:**
- ins-comp-settlement получает данные о страховках раз в сутки
- Если запрос fails → данные теряются до следующего дня
- Нет инкрементальной обработки

**Риски:**
- Задержка в взаиморасчётах на 24 часа при сбое
- Большие batch запросы нагружают систему
- Риск потери данных

**Текущее влияние:**
- Задержки в получении агентских премий
- Конфликты с бухгалтерией
- Неэффективная batch обработка

## Выводы

### Текущая архитектура не готова к росту

Основные причины:
1. Синхронная модель не масштабируется
2. Отсутствие механизмов отказоустойчивости
3. Тесная связанность компонентов
4. Неэффективное использование ресурсов

### Необходимые изменения:

1. **Переход на Event-Driven Architecture**
   - Асинхронная обработка событий
   - Loose coupling между сервисами
   - Масштабируемость

2. **Внедрение Event Streaming**
   - Apache Kafka / Yandex Data Streams
   - Гарантированная доставка событий
   - История изменений (event log)

3. **Применение паттерна Transactional Outbox**
   - Атомарность операций с БД и публикации событий
   - Гарантия eventually consistent состояния

4. **Улучшение отказоустойчивости**
   - Circuit breakers
   - Fallback механизмы
   - Retry policies

## Рекомендации к проектированию TO-BE

### Взаимодействия для перевода на Event-Streaming:

1. **ins-product-aggregator → core-app**
   - Заменить: polling каждые 15 минут
   - На: публикация событий `ProductUpdated`, `TariffChanged`
   - Выгода: real-time обновления, нет лишних запросов

2. **ins-product-aggregator → ins-comp-settlement**
   - Заменить: polling раз в сутки
   - На: подписка на события продуктов
   - Выгода: актуальные данные, нет batch запросов

3. **core-app → ins-comp-settlement**
   - Заменить: batch запрос раз в сутки
   - На: публикация событий `InsurancePolicyCreated`
   - Выгода: real-time обработка, инкрементальность

### Что НЕ меняем:

- Взаимодействие с клиентами (синхронный REST/Web)
- Взаимодействие с партнёрами (синхронный REST API)
- Запросы к БД (синхронные)
- Декомпозицию функциональности между сервисами
