# Задание 3 - Переход на Event-Driven архитектуру

## Краткое описание решения

### Спроектированное решение (TO-BE)

**Event-Driven Architecture**
- Event Streaming Platform (Apache Kafka / Yandex Data Streams)
- Асинхронная публикация и обработка событий
- Loose coupling между сервисами

**Переведено на Event-Streaming**

| Взаимодействие | Было (REST) | Стало (Events) |
|----------------|-------------|----------------|
| ins-product-aggregator → core-app | Polling каждые 15 мин | События ProductUpdated, TariffChanged |
| ins-product-aggregator → ins-comp-settlement | Polling раз в сутки | События ProductUpdated, TariffChanged |
| core-app → ins-comp-settlement | Batch запрос раз в сутки | События InsurancePolicyCreated |

**Transactional Outbox Pattern**
- Используется в core-app для публикации InsurancePolicyCreated
- Гарантирует exactly-once semantics
- Реализация через Debezium CDC (рекомендуется) или Polling

**Остается синхронным (без изменений)**
- Web App → core-app (REST) - требуется немедленный ответ пользователю
- Partners → core-app (REST API) - требования партнеров
- core-app → client-info (REST) - простое CRUD
- Все → DB (TCP/SQL) - ACID транзакции

## Ключевые компоненты

### Event Streaming Platform

**Технология:** Yandex Data Streams (Managed Kafka-compatible service)

**Topics:**
```
1. product.updates 
2. tariff.changes 
3. insurance.policy.created 
4. insurance.policy.updated 
```

### Transactional Outbox

Таблица outbox_events в core_db

**CDC с Debezium:**
- Читает outbox_events из WAL (Write-Ahead Log)
- Публикует события в Kafka автоматически
- Real-time (< 100ms latency)
- Exactly-once semantics

## Преимущества решения

### 1. Асинхронность
- Нет блокировок при вызовах к внешним системам
- ins-product-aggregator может опрашивать 10+ компаний параллельно
- События публикуются немедленно

### 2. Loose Coupling
- Сервисы не зависят друг от друга напрямую
- Можно обновлять/перезапускать сервисы независимо
- Добавление новых consumers не требует изменений producers

### 3. Real-time обновления
- Задержка: < 1 секунда (вместо 15 минут)
- Пользователи всегда видят актуальные данные
- Взаиморасчеты обрабатываются сразу

### 4. Масштабируемость
- Линейное горизонтальное масштабирование
- Готовность к росту от 5 до 10+ страховых компаний
- Kafka пропускная способность: 10,000+ events/sec на партицию

### 5. Надежность
- Guaranteed delivery через Kafka
- Transactional Outbox для атомарности
- Автоматический retry при сбоях
- Event log как source of truth

### 6. Audit и история
- Все события сохраняются в Kafka (retention 30-90 дней)
- Можно replay событий при необходимости
- Полная история изменений для регуляторных требований

## Обоснование Transactional Outbox

### Почему используем для core-app:

✅ Атомарность: запись в БД и событие - одна транзакция  
✅ Exactly-once принцип  
✅ Надежность: нет потери событий  
✅ Audit trail: таблица outbox = журнал событий  

### Почему НЕ используем для ins-product-aggregator:

- Данные о продуктах не критичны
- Можно повторно запросить у страховых компаний
- At-least-once + idempotency достаточно
- Consumer (core-app) игнорирует дубликаты по version
