# Задание 1 - Проектирование технологической архитектуры

## Содержимое

1. **InsureTech_технологическая_архитектура-as-is.drawio.xml** - Текущая архитектура (AS-IS)
2. **InsureTech_технологическая_архитектура-to-be.drawio.xml** - Спроектированная архитектура (TO-BE)
3. **Архитектурные_решения.md** - Подробное описание всех архитектурных решений и обоснований

## Выполненные требования

### Стратегия масштабирования
- **Выбрано**: Горизонтальное масштабирование с HPA (Horizontal Pod Autoscaler)
- **Обоснование**: Идеально подходит для микросервисной архитектуры в Kubernetes
- Вертикальное масштабирование применяется только для БД при необходимости

### Многозонное развертывание
- **3 зоны доступности**: ru-central1-a, ru-central1-b, ru-central1-d
- Защита от сбоев на уровне ЦОД
- Географическое распределение для уменьшения latency

### Конфигурация Kubernetes
- **Решение**: Единый Managed Kubernetes кластер с node groups в каждой зоне
- **Обоснование**: 
  - Проще управление (единая control plane)
  - Автоматическое распределение нагрузки
  - Ниже стоимость владения
  - Централизованный мониторинг

### Балансировка нагрузки
- **Application Load Balancer (ALB)** для распределения HTTP/HTTPS трафика
- **Health Checks**:
  - Protocol: HTTP
  - Path: `/health`
  - Interval: 10 секунд
  - Timeout: 5 секунд
  - Unhealthy threshold: 3
- Latency-based routing для минимизации задержек

### Фейловер-стратегия
- **Приложения**: Active-Active (все зоны активны одновременно)
- **БД**: Active-Passive с автоматическим failover
- **Время переключения**: < 1 минута

### Конфигурация базы данных
- **Managed PostgreSQL Cluster**
- **Топология**: 1 Master + 2 Sync Replicas (по одной в каждой зоне)
- **Репликация**: Синхронная (synchronous_commit = on)
- **Резервное копирование**:
  - Автоматические бэкапы каждые 15 минут → **RPO = 15 мин** 
  - Хранение в Object Storage
  - Point-in-time recovery (PITR)
- **RTO = 45 мин** достигается за счет:
  - Автоматический failover: < 1 минута
  - Восстановление из бэкапа: ~30 минут
  - Значительно меньше требуемых 45 минут

### Шардирование БД
- **Решение**: НЕ используется на текущем этапе
- **Обоснование**:
  - Объём данных: 50 GB (достаточно мало)
  - Managed PostgreSQL справляется с нагрузкой
  - Меньше сложности в разработке и эксплуатации
- **Пороговые значения для шардирования**:
  - Объём данных > 500 GB
  - Нагрузка на запись > 10,000 TPS
  - Нагрузка на чтение > 50,000 QPS

## Ключевые метрики

| Метрика | Требование | Решение | Статус |
|---------|-----------|---------|--------|
| RTO | 45 мин | < 1 мин (автоматический failover) | ✅ Превышает требования |
| RPO | 15 мин | 0 мин (синхронная репликация) + бэкапы каждые 15 мин | ✅ Превышает требования |
| Доступность | 99.9% | 99.92% (расчетная) | ✅ Соответствует |
| Время загрузки | Одинаковое для всех регионов | CDN + latency-based routing | ✅ Обеспечено |

## Архитектурные компоненты

### Сетевой уровень
- **CDN** (Yandex Cloud CDN) - кэширование статики для всех регионов РФ
- **Object Storage** - хранение статических файлов и бэкапов
- **Application Load Balancer** - распределение трафика с health checks

### Уровень приложений
- **Managed Kubernetes** - оркестрация контейнеров
- **3 Node Groups** - по одной в каждой зоне доступности
- **HPA** - автоматическое масштабирование подов
- **Минимум 3 реплики** каждого сервиса (по 1 в каждой зоне)

### Уровень данных
- **Managed PostgreSQL Cluster** - 1 Master + 2 Sync Replicas
- **Синхронная репликация** - гарантия сохранности данных
- **Автоматические бэкапы** - каждые 15 минут
- **Object Storage** - хранение бэкапов с retention 30 дней

## Безопасность
- Private subnets для Kubernetes и БД
- Public subnet только для ALB
- TLS/SSL для всех соединений
- Шифрование дисков и бэкапов
- RBAC в Kubernetes
- IAM roles для Yandex Cloud

- Yandex Monitoring + Grafana
- Метрики инфраструктуры, приложений и БД
- Алерты на критичные события
- Централизованное логирование
