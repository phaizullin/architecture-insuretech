# InsureTech - Архитектурные решения (TO-BE)

- **RTO**: 45 минут
- **RPO**: 15 минут  
- **Доступность**: 99.9%
- **Одинаковое время загрузки** для пользователей из всех регионов РФ

---

## 1. Стратегия масштабирования

### Выбрано: Горизонтальное масштабирование

**Обоснование:**
- Микросервисная архитектура идеально подходит для горизонтального масштабирования
- Kubernetes с HPA (Horizontal Pod Autoscaler) автоматически управляет количеством реплик
- Возможность масштабирования каждого сервиса независимо
- Лучшая отказоустойчивость (отказ одного пода не влияет на работу системы)

**Конфигурация HPA:**
```yaml
Метрики триггера:
- CPU > 70%
- Memory > 80%
- Custom metrics: RPS per pod > 100

Настройки реплик:
- Минимум: 3 (по 1 реплике в каждой зоне)
- Максимум: 30 (по 10 реплик в каждой зоне)
```

**Вертикальное масштабирование:**
- Применяется **только для БД** при необходимости увеличения ресурсов master-ноды
- Managed PostgreSQL позволяет изменять конфигурацию без простоя

## 2. Многозонное развертывание

### Использовано: 3 зоны доступности Yandex Cloud

**Зоны:**
- `ru-central1-a` (основная)
- `ru-central1-b` (вторичная)
- `ru-central1-d` (вторичная)

**Обоснование:**
- Защита от сбоев на уровне ЦОД
- Обеспечение доступности 99.9%
- Географическое распределение для уменьшения latency
- 3 зоны — минимум для кворума в distributed systems

---

## 3. Конфигурация Kubernetes

### Решение: Единый Managed Kubernetes кластер с node groups в каждой зоне

**Обоснование:**

**Плюсы выбранного подхода:**
- Единая control plane — проще управление
- Автоматическое распределение нагрузки между зонами
- Kubernetes scheduler автоматически размещает поды с учетом anti-affinity
- Централизованный мониторинг и логирование
- Проще обновления и деплой
- Ниже стоимость владения

**Конфигурация node groups:**
```yaml
Node Group A (ru-central1-a):
  - min_size: 3
  - max_size: 20
  - 4 vCPU, 16 GB RAM
  - auto_scale: true

Node Group B (ru-central1-b):
  - min_size: 3
  - max_size: 20
  - 4 vCPU, 16 GB RAM
  - auto_scale: true

Node Group D (ru-central1-d):
  - min_size: 3
  - max_size: 20
  - 4 vCPU, 16 GB RAM
  - auto_scale: true
```

**Anti-affinity правила:**
```yaml
# Гарантия распределения подов по разным зонам
podAntiAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
  - topologyKey: topology.kubernetes.io/zone
    labelSelector:
      matchLabels:
        app: core-app
```

## 4. Балансировка нагрузки

### Application Load Balancer (ALB)

**Функции:**
- Распределение HTTP/HTTPS трафика между зонами
- Терминация SSL/TLS
- Маршрутизация запросов к соответствующим сервисам
- Latency-based routing (минимизация задержек)

**Health Checks:**
```yaml
Health Check конфигурация:
  - Protocol: HTTP
  - Path: /health
  - Port: 8080
  - Interval: 10 секунд
  - Timeout: 5 секунд
  - Healthy threshold: 2
  - Unhealthy threshold: 3
```

**Явные health checks на схеме:**
1. ALB - Pod (каждый сервис)
   - `GET /health` каждые 10 секунд
2. Kubernetes liveness probe
   - `GET /health/live` каждые 30 секунд
3. Kubernetes readiness probe
   - `GET /health/ready` каждые 10 секунд

**Распределение трафика:**
- Алгоритм: Round-robin с учетом latency
- При отказе одной зоны трафик автоматически перенаправляется в оставшиеся зоны
- Sticky sessions для B2B клиентов

## 5. Фейловер-стратегия

### Уровень: Клиент - Приложение

**Стратегия: Active-Active для приложений**

**Сценарий отказа зоны:**
```
1. Пользователь - ALB (HTTPS запрос)
2. ALB проверяет health checks всех зон
3. Зона A: недоступна
4. Зона B: здорова
5. Зона D: здорова
6. ALB автоматически направляет трафик в зоны B и D
7. Время переключения: < 10 секунд
```

**Стратегия: Active-Passive для БД**

**Сценарий отказа master БД:**
```
1. Master PostgreSQL (зона A): недоступен
2. Managed PostgreSQL обнаруживает отказ: < 30 секунд
3. Автоматическое повышение replica (зона B) до master: < 30 секунд
4. Обновление DNS записи: < 5 секунд
5. Приложения переключаются на новый master: < 5 секунд
6. Общее время failover: < 1 минута
```

**Соответствие RTO = 45 мин:**
- Автоматический failover: < 1 минута
- Ручное восстановление из бэкапа (worst case): 30 минут
- Общее время восстановления: значительно меньше 45 минут

## 6. Конфигурация базы данных

### Managed PostgreSQL Cluster

**Топология:**
- **1 Master** (ru-central1-a) - запись и чтение
- **2 Sync Replicas** (ru-central1-b, ru-central1-d) - чтение и failover

**Репликация:**
```
Тип: Синхронная репликация (synchronous_commit = on)
- Master → Replica 1: sync (0 потерь данных)
- Master → Replica 2: sync (0 потерь данных)
```

**Преимущества синхронной репликации:**
- Гарантия сохранности данных (RPO = 0)
- Автоматический failover без потерь
- Соответствие требованиям RPO = 15 мин

### Резервное копирование

**Автоматические бэкапы:**
```yaml
Частота: Каждые 15 минут
Метод: Incremental backups + WAL archiving
Хранение: Object Storage (S3-совместимое)
Retention: 30 дней
Point-in-time recovery (PITR): Да
```

**Обеспечение RPO = 15 мин:**
1. **Основной механизм**: Синхронная репликация (RPO = 0)
2. **Дополнительный**: Бэкапы каждые 15 минут (если откажут все реплики)

**Обеспечение RTO = 45 мин:**
1. **Автоматический failover**: < 1 минута
2. **Восстановление из бэкапа** (worst case):
   - Загрузка бэкапа из Object Storage: 10 минут
   - Восстановление данных (50GB): 15 минут
   - Применение WAL логов: 5 минут
   - Всего: ~30 минут

## 7. Шардирование БД

### Решение: НЕ используется на текущем этапе

**Обоснование:**
1. **Объём данных**: 50 GB — легко умещается в одну БД
2. **Производительность**: Managed PostgreSQL справляется с нагрузкой
3. **Простота**: Меньше сложности в разработке и операционной поддержке
4. **Масштабируемость**: При необходимости можем вертикально масштабировать БД

**Пороговые значения для шардирования:**
- Объём данных > 500 GB
- Нагрузка на запись > 10,000 TPS
- Нагрузка на чтение > 50,000 QPS (при условии, что read replicas не справляются)

**Будущая стратегия шардирования (если потребуется):**
```
Шардирование по region_id (регион клиента):
- Shard 1: Центральный ФО (ru-central1-a)
- Shard 2: Северо-Западный, Южный ФО (ru-central1-b)
- Shard 3: Сибирь, Дальний Восток (ru-central1-d)

Преимущества:
- Данные клиентов хранятся ближе к ним географически
- Меньше latency
- Естественное распределение нагрузки
```

```yaml
Origin: Object Storage (S3)
Cache rules:
  - Static assets (*.js, *.css, *.png, *.jpg): 24 часа

Edge locations: Все регионы РФ (автоматически)
```

**Объект Storage:**
- Хранение статических файлов
- Хранение бэкапов БД

## 10. Мониторинг и наблюдаемость

### Yandex Monitoring + Grafana

**Метрики:**
```yaml
Инфраструктура:
  - Доступность зон
  - Состояние node groups
  - Использование ресурсов (CPU, Memory, Disk)

Приложения:
  - RPS (requests per second)
  - Latency (p50, p95, p99)
  - Error rate
  - Availability

База данных:
  - Connection pool usage
  - Query latency
  - Replication lag
  - Disk usage
```

**Алерты:**
```yaml
Critical:
  - Зона недоступна > 1 минута
  - БД недоступна > 30 секунд
  - Error rate > 5%
  - Latency p95 > 1 секунда

Warning:
  - CPU usage > 80%
  - Memory usage > 85%
  - Disk usage > 80%
  - Replication lag > 30 секунд
```
